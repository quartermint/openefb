  Rewrite tools/nasr-importer/Sources/main.swift to parse real FAA NASR 28-day subscription data files instead of generating hardcoded seed data. The importer must produce an aviation.sqlite with 20,000+ airports.

  **Data Source Strategy:**
  The FAA NASR 28-day subscription distributes data as fixed-width text files. The importer should parse these files:
  - `APT.txt` — Airport facility records (fixed-width, 1531 chars/record). Each line starting with "APT" is an airport base record.
  - `RWY.txt` — Runway records. Each line starting with "RWY" is a runway record linked by Site Number.
  - `TWR.txt` — Tower/frequency records. TWR3 records contain frequency data linked by Site Number.
  - `NAV.txt` — Navaid records (NAV1 record type). Fixed-width with identifier, type, name, lat, lon, freq.

  **Download step:** Add a `--download` flag that fetches the current NASR subscription ZIP from `https://nfdc.faa.gov/webContent/28DaySub/` (the current cycle link), extracts the needed .txt files to a temp directory, then parses them. Also support `--data-dir <path>` for offline use with pre-downloaded files.

  **If NASR fixed-width parsing proves too complex within context budget**, use this fallback approach: Download the OurAirports CSV dataset from `https://ourairports.com/data/airports.csv` (contains ~23K US airports with ICAO, name, lat, lon, elevation, type, municipality) plus `https://ourairports.com/data/runways.csv` and `https://ourairports.com/data/airport-frequencies.csv` and `https://ourairports.com/data/navaids.csv`. OurAirports data is derived from FAA sources and is adequate for Phase 1. The CSV format is trivial to parse without external dependencies.

  **Preferred approach: OurAirports CSV** (simpler, well-documented, FAA-derived, CSV format):
  The importer should:
  1. Accept `--download` to fetch CSVs from ourairports.com/data/ (airports.csv, runways.csv, airport-frequencies.csv, navaids.csv)
  2. Accept `--data-dir <path>` for offline use with pre-downloaded CSVs
  3. Parse airports.csv: filter iso_country="US", use ident as icao, name, latitude_deg, longitude_deg, elevation_ft, type (filter to large_airport, medium_airport, small_airport, seaplane_base — skip closed, heliport unless public), municipality for city, gps_code or local_code for faa_id
  4. Parse runways.csv: JOIN on airport_ref (airport id), extract length_ft, width_ft, surface, lighted, le_ident (base end), he_ident (reciprocal end), le_latitude_deg, le_longitude_deg, he_latitude_deg, he_longitude_deg
  5. Parse airport-frequencies.csv: JOIN on airport_ref, extract type, description, frequency_mhz
  6. Parse navaids.csv: filter iso_country="US", extract ident, type, name, latitude_deg, longitude_deg, frequency_khz (convert to MHz for VOR/VORTAC), elevation_ft, magnetic_variation_deg

  <action>
  Fix the MapView.updateUIView method to only call `mapViewModel.onFirstLocationReceived(location:)` once per map lifecycle, rather than on every SwiftUI update cycle after the first GPS fix.

  Current code (lines 67-69):
  ```swift
  if mapViewModel.appState.firstLocationReceived {
      mapViewModel.onFirstLocationReceived(location: position)
  }
  ```

  This is called every time SwiftUI re-renders MapView (any @Observable change triggers updateUIView). The MapViewModel guards internally with `hasAnimatedToFirstLocation`, so behavior is correct but the repeated calls are wasteful.

  Fix: Add a static-like tracking mechanism. Since UIViewRepresentable does not support @State in the struct itself, use the Coordinator to track this:

  1. Add a property to the existing Coordinator class:
     ```swift
     var hasCalledFirstLocation: Bool = false
     ```

  2. Change updateUIView to check the coordinator flag:
     ```swift
     func updateUIView(_ mapView: MLNMapView, context: Context) {
         // Sync ownship position
         if let position = mapViewModel.appState.ownshipPosition {
             let heading = mapViewModel.appState.track
             mapService.updateOwnship(location: position, heading: heading)

             // Handle first location animation (fire once only)
             if mapViewModel.appState.firstLocationReceived && !context.coordinator.hasCalledFirstLocation {
                 context.coordinator.hasCalledFirstLocation = true
                 mapViewModel.onFirstLocationReceived(location: position)
             }
         }

         // Sync layer visibility
         for layer in MapLayer.allCases {
             let visible = mapViewModel.appState.visibleLayers.contains(layer)
             mapService.setLayerVisibility(layer, visible: visible)
         }

         // Sync sectional opacity
         mapService.setSectionalOpacity(mapViewModel.appState.sectionalOpacity)
     }
     ```

  This ensures onFirstLocationReceived is called exactly once, at the Coordinator level which persists across SwiftUI update cycles.
  </action>
